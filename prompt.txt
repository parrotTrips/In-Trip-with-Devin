## Contexto do projeto

Estou desenvolvendo o Parrot Trips, um aplicativo web para viajantes que inclui autenticação via WhatsApp OTP e gerenciamento de dados de viagem. O backend é FastAPI (Python) e o frontend é React + TypeScript. O banco de dados é PostgreSQL hospedado no Supabase.

---

## O que já foi implementado e validado

### Autenticação: WhatsApp OTP + JWT

O sistema de autenticação está completamente implementado e testado end-to-end:

1. O usuário digita o telefone no frontend
2. O backend gera um código OTP de 6 dígitos, salva no banco e envia via WhatsApp Cloud API (Meta)
3. O usuário digita o código no app
4. O backend valida o OTP, cria o usuário se for o primeiro acesso, e retorna um JWT assinado (HS256, válido por 14 dias)
5. O frontend armazena `{ userId, phone, name, token }` no localStorage
6. Todas as chamadas de API incluem `Authorization: Bearer <token>`
7. O backend valida o token via middleware global (FastAPI `BaseHTTPMiddleware`) em todas as rotas exceto `/auth/*` e `/healthz`

O teste E2E passou com sucesso em todos os 6 passos:
- OTP entregue via WhatsApp no celular
- JWT gerado e retornado
- Rota protegida sem token → 401
- Rota protegida com token válido → passou
- Token inválido → 401

### Limitação atual: conta sandbox da Meta

A conta de WhatsApp usada nos testes é o **sandbox de teste da Meta** ("Test WhatsApp Business Account"):

- `WHATSAPP_PHONE_NUMBER_ID=1085386851333264` — número de teste da Meta
- Só envia para números adicionados manualmente como destinatários de teste
- **Não tem permissão para criar templates próprios** — tentamos via API e via dashboard, ambos retornaram erro de permissão
- O access token gerado pela tela "Try it out" expira em ~24h
- O template em uso (`jaspers_market_order_confirmation_v1`) é um template de exemplo da Meta usado como workaround temporário — o código OTP aparece no campo "número do pedido" da mensagem

### Template desejado para produção

O template final se chama `intripauth` e deve ser do tipo **Authentication** no idioma **pt_BR**. O backend já está preparado para ele: a função `_build_template_components()` em `backend/app/services/auth_service.py` usa o payload correto automaticamente quando `WHATSAPP_TEMPLATE_NAME=intripauth`.

O template `intripauth` deve ter:
- Corpo com `{{1}}` = código OTP
- Botão URL com `{{1}}` = código OTP (deep link de verificação)
- Idioma: pt_BR

### Restrições e requisitos

- O access token **não deve ser commitado** no git — só fica no `.env` local (já está no `.gitignore`)
- O número de telefone do WhatsApp Business não precisa ser o mesmo usado para receber OTPs — é o número remetente
- O template `intripauth` precisa ser aprovado pela Meta antes de poder ser usado — templates Authentication costumam ser aprovados automaticamente ou muito rápido
- O público alvo são viajantes brasileiros, então pt_BR é o idioma correto
- Volume esperado: 50 a 200 usuários por mês — dentro do free tier de 1.000 conversas/mês da Meta
