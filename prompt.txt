## Projeto: Parrot Trips — In-Trip App

App web para viajantes acompanharem uma experiência de viagem. Backend FastAPI (Python), frontend React + TypeScript, banco PostgreSQL no Supabase.

---

## O que foi implementado e está funcionando

### Autenticação WhatsApp OTP + JWT
- Fluxo completo: telefone → OTP via WhatsApp → JWT de 14 dias
- Template `intripauth` (Authentication, idioma `en`) aprovado e ativo na Meta
- Apenas telefones cadastrados na tabela `users` conseguem pedir OTP (403 para não autorizados)
- JWT contém: `sub` (user_id), `phone`, `role`, `exp`

### Dois tipos de usuário
- `role = 'traveler'` → vê o mapa da jornada da viagem (tela do viajante)
- `role = 'staff'` → vê a tela operacional de tarefas do dia (tela staff)
- O roteamento acontece em `frontend/src/app/App.tsx` baseado no `user.role`

### Banco de dados — tabelas existentes
- `users` — id, phone, full_name, email, status, role (`traveler` | `staff`)
- `trips` — viagens
- `trip_travelers` — vínculo usuário ↔ viagem (para viajantes)
- `trip_staff` — vínculo usuário ↔ viagem (para staff, com campo `function`)
- `staff_tasks` — tarefas operacionais do staff por fase/dia da viagem
- `trip_phases` — fases e dias da viagem
- `trip_activities` — atividades de lazer do viajante
- `traveler_profiles`, `traveler_products` — dados do viajante
- `traveler_checklist_progress`, `traveler_phase_progress` — progresso

### Frontend — telas existentes
- `LoginScreen` — login WhatsApp OTP
- `HomeScreen` — mapa da jornada do viajante (dados mockados, não conectado ao backend)
- `PhaseDetails` — detalhes de uma fase pré-viagem (parcialmente conectado)
- `DayDetails` — detalhes de um dia in-trip (mockado)
- `ProfileScreen` — perfil do viajante (conectado ao backend)
- `StaffScreen` — tela operacional do staff (mockada, estrutura visual pronta)

## Dev auto-login (para desenvolver sem passar pelo OTP)

Descomente no `frontend/.env`:
```env
VITE_DEV_AUTO_LOGIN=true
VITE_DEV_USER_ID=00000000-0000-0000-0000-000000000001
VITE_DEV_USER_PHONE=+5500000000000
VITE_DEV_USER_NAME=Viajante Teste
VITE_DEV_USER_ROLE=traveler
```
Troque `VITE_DEV_USER_ROLE=traveler` por `staff` para ver a tela do staff.

## Próximos passos (ordem sugerida)

### 1. Conectar HomeScreen ao backend (RF02 + RF03)
A tela do viajante (`HomeScreen`) ainda usa dados mockados em `frontend/src/features/trip/data/tripData.ts`. Precisa:
- Criar endpoint no backend: `GET /trip/active?user_id=...` → retorna a viagem ativa do viajante
- Criar endpoint: `GET /trip/{trip_id}/phases` → retorna as fases/dias com progresso do viajante
- Conectar o frontend a esses endpoints

### 2. Conectar StaffScreen ao backend
A tela do staff também usa dados mockados. Precisa:
- Criar endpoint: `GET /staff/trip/{trip_id}/tasks` → retorna as tarefas por dia
- Conectar `StaffScreen` a esse endpoint

### 3. Adicionar dados de viagem no banco
Hoje o banco tem as tabelas mas sem dados reais. Precisa inserir:
- Uma trip de exemplo no Supabase
- Vincular Marcelo e Vitor como `trip_staff` nessa trip
- Inserir algumas `staff_tasks` para ver a tela do staff com dados reais

### 4. Deploy
- Backend: Railway, Render ou Fly.io
- Frontend: Vercel ou Netlify
- Atualizar `VITE_API_URL` no frontend para a URL de produção
